## 新增规则

### 1. 代码质量标准 (coding-quality-standards.md)

**优先级**: 🔴 CRITICAL

**内容**:
- 10 大核心代码质量原则
- 每个原则包含：规则说明、错误示例、正确示例、检查清单
- 命名规范（Python + TypeScript）
- 工具链配置
- Code Review 检查清单

**核心原则**:
```
可读性 > 可维护性 > 可测试性 > 健壮性 > 性能
```

**关键规则**:
1. 单一职责原则 - 函数 20-40 行，类 200-400 行
2. 早期返回 & 卫语句 - 减少嵌套
3. 显式优于隐式 - 避免魔法
4. 防御性编程 - 永远不要相信外部输入
5. DRY 但不过度抽象 - 三次重复才抽象
6. 错误处理要清晰 - 分层异常
7. 可测试性 - 依赖注入
8. 注释写"为什么" - 不写"做什么"
9. 保持代码"活" - 删除僵尸代码
10. 其他高频规则 - 魔法值、嵌套、日志等

### 2. AI 开发效率规范 (ai-development-efficiency.md)

**优先级**: 🟡 HIGH

**内容**:
- 11 大 AI 开发效率技巧
- Token 节省技巧表格
- 结构化 Prompt 模板
- **Kiro Spec 工作流**（新增）
- 实战检查清单

**核心原则**:
```
短而精的上下文 > 长而全的上下文
```

**黄金法则**:
1. 让 AI 每次只做一件事
2. 把大任务拆解成可独立验证的小步
3. 主动管理上下文窗口
4. 明确输出格式和约束
5. **使用 Spec 工作流：让 AI "观全局，做细节"**（新增）

**关键技巧**:
- 拆分大任务 - 小步快跑
- 短上下文模式 - 只给必要信息
- 主动总结 - 压缩历史（节省 40-80% Token）
- 结构化 Prompt - 角色+目标+格式+约束
- 区分探索型和执行型 - 避免混用
- 使用模板库 - 节省时间和 Token
- 标记重点 - "只改这里"
- Kiro 上下文管理 - #File, #Folder, #Problems
- 避免重复输出 - 只返回改动
- **Spec 工作流 - 观全局，做细节（节省 70-80% Token）**（新增）

**Spec 工作流优势**:
- 上下文复用：Spec 文件可被多个任务引用
- 全局视野：AI 理解完整需求，不偏离方向
- 细节聚焦：每次只做一个小任务，出错率低
- 进度可控：任务列表清晰，随时知道进度
- 易于回滚：任务独立，出错时只需回滚一个任务

### 3. AI 工作流模板库 (.kiro/templates/)

**优先级**: 🟡 HIGH

**创建了 8 个常用模板**:

1. **optimize-function.md** - 优化函数性能/可读性/可维护性
2. **write-unit-test.md** - 编写单元测试（覆盖正常+边界+异常）
3. **refactor-code.md** - 重构代码（应用设计原则）
4. **add-type-annotations.md** - 添加完整类型注解
5. **fix-bug.md** - 修复 Bug（包含根因分析）
6. **add-logging.md** - 添加结构化日志
7. **extract-function.md** - 提取函数（单一职责）
8. **spec-workflow-guide.md** - Spec 工作流快速指南（新增）

**使用方法**:
```
1. 从 .kiro/templates/ 复制模板
2. 替换 [占位符] 为实际内容
3. 粘贴到 Kiro 对话框
4. 节省 50-90% 的 Prompt 编写时间
```

## 规则体系结构

### 优先级分类

```
🔴 CRITICAL（关键规则）
├── coding-quality-standards.md
└── async-sync-safety.md

🟡 HIGH（高优先级规则）
├── ai-development-efficiency.md
├── typescript-export-rules.md
├── i18n-translation-rules.md
├── file-organization-rules.md
└── documentation-minimalism-rules.md

🟢 MEDIUM（中优先级规则）
├── product.md
├── structure.md
└── tech.md

🔵 INFO（信息类）
└── auto-approve-guide.md
```

### 规则冲突解决

```
优先级: CRITICAL > HIGH > MEDIUM > INFO
范围: 项目特定规则 > 通用规则
时间: 新规则 > 旧规则（相同主题时）
```

## 预期效果

### 代码质量提升

- **可读性**: 函数更短、嵌套更少、命名更清晰
- **可维护性**: 单一职责、清晰的错误处理
- **可测试性**: 依赖注入、纯函数
- **健壮性**: 防御性编程、输入校验

### 开发效率提升

- **AI 交互效率**: 节省 40-80% 的 Token
- **Prompt 编写时间**: 节省 50-90%（使用模板）
- **上下文管理**: 更精准、更高效
- **错误率降低**: 明确的约束和格式

### 文档质量提升

- **文档数量**: 减少 50-70% 的冗余文档
- **文档质量**: 更聚焦、更有用
- **维护成本**: 降低 60-80%

## 总结

1. **代码质量标准** - 确保代码可读、可维护、可测试

这些规则基于高级程序员的最佳实践，经过实战验证，可以显著提升团队的开发效率和代码质量。
