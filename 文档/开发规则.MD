步骤1: 创建 Steering 文件结构

在项目根目录下，确保 .kiro/steering/ 存在（如果没有，创建它）。
在里面创建 Markdown 文件，每个文件专注一个主题（最小可用：从1-2个文件起步）。文件名为小写下划线分隔，如 coding_standards.md、documentation_rules.md。
Steering 文件会自动被 Kiro AI 加载，用于任务指导、代码审查和提示生成。例如，当你创建任务时，AI 会参考这些规则自动提示“请遵守 snake_case 命名”。

# 编码规范规则（Coding Standards）

- 文件/目录：全小写下划线，如 `kiro_utils.py`。
- 常量：全大写下划线，如 `DEFAULT_LANGUAGE = "zh"`。

- **加载规则**：Kiro 会自动扫描 `.kiro/steering/` 中的 Markdown 文件。重启 Kiro 或在聊天中输入 `/reload steering`（如果支持）。

### 步骤4: 高级优化与最小可用实践
- **最小可用（MVP）**：先创建1个文件（如 coding_standards.md），测试在小任务中是否自动提示规则。
- **减少随机性**：用 Git Hooks 强制格式化（pre-commit install Black/Ruff）。
- **扩展提示**：如果想更智能，在 steering 文件中添加“AI 提示模板”，如“当代码不合规时，回复：'违反规则 X，请修正为 Y'”。
- **测试**：创建一个任务“写一个不合规的函数”，观察 AI 是否提示修正。
- **潜在问题**：如果规则太多，AI 可能忽略部分——迭代精简。安全起见，避免规则中包含执行命令。

以下按照重要性与出现频率排序（主观排序），并附上为什么高级程序员特别重视它们，以及常见的实际做法。
1. 单一职责原则（Single Responsibility Principle） & 函数/类规模控制

核心理念：一个函数/类只做一件事，且只因为一个原因改变。
常见硬性指标：
函数尽量控制在 20–40 行（最理想 10–20 行）
类尽量控制在 200–400 行（视语言而定）

为什么高级程序员在意：超过这个规模的函数几乎一定隐藏了多个职责，未来改动时极易引入回归 bug。
常见做法：
发现函数开始出现第二层缩进 → 考虑抽取
发现同一个函数里有“读数据 → 处理 → 写数据 → 发通知” → 拆分成 4 个小函数


2. 早期返回 & 卫语句（Guard Clauses）

写法对比：

Python# 差
if condition:
    # 核心逻辑 50 行
    ...
else:
    return None

# 好（卫语句）
if not condition:
    return None

# 核心逻辑（缩进减少，阅读顺序更自然）
...

高级程序员偏好：把所有异常/无效情况提前返回，核心逻辑保持最浅的缩进层级。
好处：阅读时不用脑补“如果不满足条件会怎样”，认知负担大幅降低。

3. 显式优于隐式（Explicit is better than implicit）

Python 禅宗名言，也是很多高级程序员的信条。
常见体现：
不要依赖魔法方法、隐式转换、默认值、框架黑魔法
函数参数不要用太多默认值（尤其是可变默认值！）
返回值类型不一致时宁可抛异常也不要返回 None/空字符串/空列表混用
配置优先从环境变量或配置文件读取，而不是写死


4. 防御性编程 & 输入校验

核心思想：永远不要相信外部输入（包括自己上一层调用方）
常见做法：
函数开头做参数校验（类型、范围、非空、格式等）
使用断言（assert）或抛出明确的业务异常
对所有外部数据（API、数据库、文件、用户输入）都做校验

高级程序员常说的一句话：“我宁可早抛异常早崩溃，也不愿意晚点出现诡异的行为。”

5. DRY 但不过度抽象（Don't Repeat Yourself — but don't be too DRY）

高级思维：重复三遍以上才考虑抽取抽象；两次重复可以接受；一次不抽取。
常见陷阱：为了消除两处重复代码，创建了一个非常复杂的工具类/高阶函数，反而降低了可读性。
推荐做法：
先复制粘贴 + 小改
第二次复制时思考是否真的值得抽象
第三次才动手重构


6. 错误处理要清晰、有层次

优秀模式（由浅到深）：
卫语句提前返回（最常见）
Result / Either / Option 类型（函数式风格）
异常（业务异常要明确区分层级）

反模式：
吃掉所有异常然后返回 None
catch (Exception e) { log.error(...); } 然后继续跑
层层透传异常却不做任何处理


7. 可测试性（Testability）
高级程序员写代码时会同时在脑子里写测试。
常见设计原则：

函数尽量纯（无副作用）
依赖通过参数或构造函数注入（依赖倒置）
避免直接 new 对象、调用静态方法、获取当前时间、读写文件、发网络请求
优先写单元测试能测的代码，而不是写一堆集成测试才能覆盖的代码

8. 注释写“为什么”，而不是“做什么”

高级程序员共识：
好的代码 + 好的命名 → 注释应该很少
真正需要注释的地方是：为什么这样做、业务背景、权衡取舍、规避的坑、临时方案、TODO/XXX/HACK

反例（最常见的低级注释）：Pythoni += 1  # i 加 1

9. 保持代码“活”的状态（避免僵尸代码）

定期删除不再使用的代码（dead code）
不要注释掉一大段代码（应该进 git 历史）
避免长期存在的 TODO / FIXME（设定时限或转成 issue）

11. 更好的用 AI 开发：如何避免超长上下文 & 避免每次过度浪费 Token
高级程序员在使用 AI 辅助编程时，会把“上下文长度控制”和“Token 效率”当作核心优化目标之一，因为这直接影响开发速度、准确率和成本。
核心原则

短而精的上下文 > 长而全的上下文
让 AI 每次只做一件事，而不是一次做十件事
把大任务拆解成可独立验证的小步
主动管理上下文窗口，而不是被动堆积历史

具体实用规则与技巧（按重要程度排序）

把大任务拆成明确、独立的小步骤（最重要）
不要一次问：“帮我把整个项目重构为微服务架构”
而是拆成：
先分析当前架构的痛点
设计服务拆分边界
给出第一个服务的领域模型
写第一个服务的接口契约
实现第一个服务的基本骨架
……

好处：每一步上下文都很短，AI 出错率低，你也能及时纠错

使用“当前文件 + 相关片段 + 明确指令”的模式
坏例子（超长上下文）：text这是我整个项目的代码……（粘贴 5000 行）……请帮我优化这个函数
好例子（短上下文、高效）：text当前文件：user_service.py（已提供完整代码）

请只关注第 128–145 行这段代码：

```python
def get_user(...):
    ...问题：这个函数有性能瓶颈，且可读性差。
请给出优化建议，并只返回优化后的这段函数代码 + 必要的说明。
不要修改其他部分，不要输出整个文件。

主动总结与压缩历史上下文（Context Summarization）
当对话已经进行了 8–10 轮以上，主动让 AI 做一次总结：text请用 200 字以内总结我们目前讨论的项目现状、已完成的部分、待解决的核心问题、当前的下一步计划。
以后我都会基于这个总结继续提问。
然后把这个总结复制保存下来，作为后续对话的开头固定使用。

- 输出格式：只返回优化后的完整函数代码 + 简短的改动说明（不超过 100 字）

代码：

探索型（脑暴、方案设计）：可以允许较长的上下文，允许 AI 发散思考
执行型（写代码、改 bug、优化）：强制短上下文 + 明确输出格式

一句话总结（可以直接贴到你的 steering 文件里）
在使用 AI 辅助开发时，永远优先追求短上下文、高聚焦、明确输出格式。
大任务必须拆成小步，每步只给 AI 必要的上下文和清晰的约束。
