# 工程可溯性与 AI 上下文

**用途**：统一「规则从哪来、进度看谁、文档与代码如何对齐」，便于人工与 AI 在任意会话中快速建立可靠上下文，并留下可后查的记录入口。

**维护**：重大架构/规则/索引变更时更新本节或下表；日常小改可在 Git 提交说明中说明即可。

---

## 1. 长记忆与短记忆（面向 AI 与协作者）

| 类型 | 含义 | 本项目中的载体 |
|------|------|----------------|
| **长记忆** | 跨会话、可版本化、应以仓库为准的事实 | `.kiro/steering/CORE.md`；`.kiro/rules/` 按需规则；`.kiro/specs/<主题>/` 三件套；`.kiro/project-progress.json`；**本文档**与 **`文档/文档总索引.md`**；`.cursor/skills/superdata-kiro-dev/SKILL.md`（Cursor 侧摘要） |
| **短记忆** | 当前任务、对话、未合并改动 | 用户指令、当前分支、`git diff`、Issue/PR 描述 |

**实践要点**：

- 回答「结构、技术栈、目录约定」优先读 `.kiro/rules/structure.md`、`.kiro/rules/tech.md`，并与 `src/`、`frontend/src/` 对照。
- 回答「进度、里程碑」以 `.kiro/project-progress.json` 为线索，**与代码/PR 交叉验证**（该文件可能滞后）。
- **全局鸟瞰**：按 `CORE.md` 中的「全局分析模式」加载：`product.md`、`structure.md`、`tech.md`、`project-progress.json`、`文档/文档总索引.md`。

## 2. 真源与一致性（避免「文档滞后」或「纸上设计」）

**原则**：实现行为以**代码与可运行配置**为准；文档描述预期或约束，二者冲突时**先查清代码再改文档或改代码**，禁止在未核对的情况下按过时文档大规模改代码。

| 风险 | 建议 |
|------|------|
| Spec/设计**落后于**代码 | 以代码为准；在对应 `.kiro/specs/<主题>/` 三件套中更新设计/任务状态，或删改过时段落并注明日期。 |
| 文档**已设计**但代码**未实现** | 在三件套或 API 文档中标注 **未实现** / **规划中**，或拆成独立任务；避免正文读起来像已上线。 |
| `project-progress.json` 与事实不符 | 合并功能或发版后更新该文件；AI 引用时需提示「可能与分支不一致」。 |
| `文档/` 下专题文与代码不一致 | 在专题文顶部增加 **状态** 行：`与实现对齐 | 截至 YYYY-MM-DD` 或 `部分过时，见 CODE/路由为准`。 |

**禁止**（与 `.kiro/steering/CORE.md` 一致）：无必要地新增「报告类」长文；可溯性靠 **索引 + 三件套 + 提交记录 + 下表** 即可。
