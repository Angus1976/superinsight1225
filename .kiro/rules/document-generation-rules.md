# 文档生成规范

**Version**: 1.0  
**Last Updated**: 2026-02-04  
**Priority**: HIGH  
**加载方式**: 手动加载（按需引用）

---

## 📌 核心原则（必读）

**严格分类 > 定期更新索引 > 禁止随意生成**

所有文档必须按分类保存到 `文档/` 对应子目录，并更新文档总索引。

---

## 🎯 3 条铁律（必须遵守）

1. **禁止在根目录生成文档** - 除了 README.md
2. **必须按分类保存** - 使用 `文档/` 下的 10 个分类目录
3. **必须更新索引** - 每次生成文档后更新 `文档/文档总索引.md`

---

## ⚡ 快速参考（80% 场景够用）

### 文档分类目录

| 分类 | 目录 | 适用场景 |
|------|------|---------|
| 快速开始 | `文档/快速开始/` | 快速启动、入门指南 |
| 部署指南 | `文档/部署指南/` | 部署、安装、环境配置 |
| Docker | `文档/Docker/` | Docker 相关操作和问题 |
| 国际化翻译 | `文档/国际化翻译/` | i18n、多语言支持 |
| 问题修复 | `文档/问题修复/` | Bug 修复、问题诊断 |
| 任务完成 | `文档/任务完成/` | 任务完成报告、测试结果 |
| 状态报告 | `文档/状态报告/` | 系统状态、部署状态 |
| 执行报告 | `文档/执行报告/` | 执行过程、最终报告 |
| 开发流程 | `文档/开发流程/` | 开发规范、测试流程 |
| 索引和说明 | `文档/索引和说明/` | 索引文档、快速参考 |

### 文档命名规范

**格式**: `{功能/模块}-{类型}.md`

**示例**：
- ✅ `Docker重建指南.md`
- ✅ `管理后台翻译完成报告.md`
- ✅ `后端修复完成报告.md`
- ❌ `report.md`（太模糊）
- ❌ `fix_bug_20260204.md`（不要用日期）

### 更新索引流程

1. 生成文档后，立即更新 `文档/文档总索引.md`
2. 在对应分类下添加文档条目
3. 更新文档统计数据
4. 提交 Git 时同时提交索引文件

### 命名格式

**基本格式**: `{功能/模块}-{类型}.md`

**类型关键词**：
- 指南：guide、指南
- 报告：report、报告
- 摘要：summary、摘要
- 状态：status、状态
- 修复：fix、修复
- 完成：complete、完成
- 测试：test、测试

### 命名示例

**✅ 好的命名**：
```
Docker重建指南.md              # 清晰的功能 + 类型
管理后台翻译完成报告.md        # 明确的模块 + 状态 + 类型
后端修复完成报告.md            # 清晰的范围 + 状态 + 类型
任务5.2-控制台测试完成.md      # 任务编号 + 功能 + 状态
系统监控日期修复.md            # 功能 + 问题 + 类型
```

**❌ 不好的命名**：
```
report.md                      # 太模糊
fix.md                         # 不知道修复了什么
test_20260204.md              # 不要用日期
temp.md                        # 临时文件不应该提交
doc1.md                        # 无意义的编号
```

### 命名规则

1. **使用中文命名** - 提高可读性
2. **功能优先** - 先说功能，再说类型
3. **避免日期** - 用 Git 历史追踪时间
4. **避免缩写** - 除非是通用缩写（如 Docker、Git）
5. **避免版本号** - 用 Git 管理版本

<summary><b>规则 3: 更新索引详解</b>（点击展开）</summary>

### 更新步骤

1. **打开索引文件**
   ```bash
   # 编辑文档总索引
   vim 文档/文档总索引.md
   ```

2. **在对应分类下添加条目**
   ```markdown
   | [新文档名称.md](./分类目录/新文档名称.md) | 文档说明 | 2026-02-04 |
   ```

3. **更新统计数据**
   ```markdown
   ### 按分类统计
   
   | 分类 | 文件数 | 占比 |
   |------|--------|------|
   | 问题修复 | 9 | 14.1% |  # 从 8 增加到 9
   | **总计** | **65** | **100%** |  # 从 64 增加到 65
   ```

4. **更新文档总数**
   ```markdown
   **文档总数**: 65 个文件  # 从 64 更新到 65
   ```

### 索引模板

```markdown
### X. 分类名称 (Y 个文件)

分类说明文字。

| 文档名称 | 说明 | 更新日期 |
|---------|------|---------|
| [文档1.md](./分类/文档1.md) | 文档1说明 | 2026-02-04 |
| [文档2.md](./分类/文档2.md) | 文档2说明 | 2026-02-04 |

**推荐阅读顺序**: 文档1 → 文档2
```

<summary><b>规则 4: 禁止生成的文档</b>（点击展开）</summary>

### 禁止生成的文档类型

根据文档最小化规范，以下类型的文档**禁止生成**：

1. **诊断报告** - 应该在代码注释或 Git commit 中记录
2. **执行报告** - 应该在 tasks.md 中更新进度
3. **实现总结** - 应该在 Git commit 中记录
4. **测试指南** - 应该在 tasks.md 中简要说明
5. **测试检查清单** - 应该在代码注释中说明
6. **测试准备文档** - 应该在 tasks.md 中说明
7. **快速启动指南** - 除非是项目级别的
8. **完成总结** - 应该在 tasks.md 中更新
9. **资源索引** - 除非是项目级别的
10. **重复的代码示例** - 应该在代码中

### 替代方案

| 禁止的文档 | 替代方案 |
|-----------|---------|
| 诊断报告 | Git commit message |
| 执行报告 | tasks.md 进度更新 |
| 实现总结 | Git commit message |
| 测试指南 | tasks.md 简要说明 |
| 测试检查清单 | 代码注释 |
| 完成总结 | tasks.md 标记完成 |

### 例外情况

以下情况可以生成文档：

1. **项目级别的文档** - 如项目 README、快速启动指南
2. **重要的问题修复** - 需要详细记录的 Bug 修复
3. **重大功能实现** - 需要详细说明的功能
4. **部署和配置** - 部署指南、配置说明

</details>

<details>
<summary><b>规则 5: 文档审查流程</b>（点击展开）</summary>

### 生成文档前审查

1. **必要性审查**
   - 这个文档是必须的吗？
   - 能否合并到现有文档？
   - 是否违反文档最小化规范？

2. **分类审查**
   - 确定了正确的分类？
   - 分类目录是否存在？
   - 是否需要创建新分类？

3. **命名审查**
   - 文档名称是否清晰？
   - 是否符合命名规范？
   - 是否与现有文档重复？

### 生成文档后审查

1. **位置审查**
   - 文档是否在正确的目录？
   - 是否在根目录？（禁止）
   - 是否在 `文档/` 目录下？

2. **内容审查**
   - 内容是否超过限制？
   - 是否有冗余内容？
   - 是否符合文档规范？

3. **索引审查**
   - 是否更新了文档总索引？
   - 索引中的信息是否正确？
   - 统计数据是否更新？

### 定期审查

**每周审查**：
- 检查是否有未分类的文档
- 检查是否有重复的文档
- 检查索引是否准确

**每月审查**：
- 删除过时的文档
- 合并相似的文档
- 优化文档结构

</details>
