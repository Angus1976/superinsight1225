# AI 开发效率规范 (AI Development Efficiency)

**Version**: 2.0  
**Status**: ✅ Active  
**Last Updated**: 2026-02-04  
**Priority**: HIGH  
**加载方式**: 手动加载（按需引用）

---

## 📌 核心原则（必读）

**短而精的上下文 > 长而全的上下文**

---

## 🎯 5 条黄金法则（日常使用）

1. **每次只做一件事** - 不要一次问十件事
2. **拆小步** - 大任务拆成可验证的小步
3. **渐进式给上下文** - 先给核心问题，需要时再补充
4. **明确约束和格式** - 告诉 AI 只输出什么
5. **用 Spec 工作流** - 观全局（Spec 文件），做细节（单个任务）

---

## ⚡ 快速参考（80% 场景够用）

### Prompt 模板
```
角色：你是资深 [语言] 工程师
任务：[具体要做什么]
约束：只修改 [范围]、不引入新依赖、输出格式：[格式]
代码：[粘贴 20-50 行]
```

### 渐进式上下文（3 层）
```
第 1 层：问题 + 预期 + 约束
第 2 层：相关代码 + 错误信息（如需要）
第 3 层：项目背景（如仍需要）
```

### Spec 工作流（推荐）
```
1. 创建 Spec 三件套（requirements.md, design.md, tasks.md）
2. 执行单个任务："请执行 Task 1.1，上下文：#File:xxx/requirements.md"
3. 更新进度："Task 1.1 完成，请更新 tasks.md"
```

### Token 节省
| 做法 | 节省 |
|------|------|
| 只贴相关函数 | 50-90% |
| 只输出改动 | 30-70% |
| Spec 工作流 | 70-80% |

### 对话管理
- 超过 10 轮？→ 让 AI 总结（200 字）
- AI 出错？→ 重新描述需求 + 明确约束
- 输出格式不对？→ 明确说"只返回修改的函数"

<summary><b>1. 结构化 Prompt 详解</b>（点击展开）</summary>

### 完整模板

```
你现在是资深 [Python/TypeScript/...] 工程师，擅长 [性能优化/代码整洁/...]。

任务：[具体要做什么]

约束：
- 只修改 [具体范围]
- 不要引入新依赖
- 保持原有功能完全一致
- 输出格式：[具体格式要求]

代码/上下文：
[粘贴 20-50 行代码]
```

### 为什么有效
- **角色**：让 AI 进入专家模式
- **任务**：明确目标
- **约束**：限制发散
- **格式**：控制输出长度

### 示例

```
你是资深 Python 后端工程师，擅长性能优化。

任务：优化下面这段代码的数据库查询性能。

约束：
- 只修改数据库查询部分
- 不改变函数签名
- 输出格式：只返回修改的函数 + 简短说明（< 100 字）

代码：
def get_users():
    users = []
    for id in range(1, 100):
        user = db.query(User).filter_by(id=id).first()
        users.append(user)
    return users
```

<summary><b>2. 渐进式上下文构建详解</b>（点击展开）</summary>

### 三层模型

```
第 1 层：核心问题（必需）
├── 问题描述
├── 预期结果
└── 关键约束

第 2 层：相关上下文（按需）
├── 相关代码片段
├── 错误信息
└── 依赖关系

第 3 层：背景信息（可选）
├── 项目架构
├── 历史决策
└── 相关文档
```

### 实战示例

**第 1 轮：最小上下文**
```
问题：用户登录接口返回 500 错误
预期：返回 200 和 JWT token
约束：不要修改数据库结构
```

**第 2 轮：添加必要上下文（如果 AI 需要）**
```
相关代码：
@app.post("/api/auth/login")
def login(email: str, password: str):
    user = db.query(User).filter_by(email=email).first()
    return {"token": generate_token(user)}

错误信息：
AttributeError: 'NoneType' object has no attribute 'id'
```

**第 3 轮：添加背景信息（如果仍需要）**
```
背景：
- 使用 FastAPI + SQLAlchemy
- JWT 认证已配置
- 数据库连接正常
```

### Token 节省
- 传统方式：一次给所有信息（5000+ tokens）
- 渐进式：按需添加（500-1500 tokens）
- **节省 50-70%**

<summary><b>3. Spec 工作流详解</b>（点击展开）</summary>

### Spec 三件套

```
.kiro/specs/{feature-name}/
├── requirements.md  # 需求文档（< 5000 字）
├── design.md        # 设计文档（< 3000 字）
└── tasks.md         # 任务列表（< 2000 字）
```

### 完整流程

**阶段 1：创建 Spec（观全局）**
```
请为 [功能名称] 创建 Spec 三件套：

需求描述：
[用户故事、验收标准、优先级]

输出：
1. requirements.md - 完整的需求文档
2. design.md - 架构决策、关键接口、数据模型
3. tasks.md - 任务分解、时间估计、依赖关系
```

**阶段 2：执行任务（做细节）**
```
请执行 tasks.md 中的 Task 1.1：[任务名称]

上下文：
- 需求：#File:.kiro/specs/[feature]/requirements.md
- 设计：#File:.kiro/specs/[feature]/design.md

约束：
- 只实现这一个任务
- 遵循 design.md 中的设计
- 输出格式：只返回新增的代码
```

**阶段 3：更新进度**
```
Task 1.1 已完成，请更新 tasks.md：
- 标记为完成 ✅
- 添加实际耗时
- 如有新问题，添加到待办
```

### 优势

| 优势 | 说明 | Token 节省 |
|------|------|-----------|
| 上下文复用 | Spec 文件可被多个任务引用 | 60-80% |
| 全局视野 | AI 理解完整需求，不偏离方向 | - |
| 细节聚焦 | 每次只做一个小任务，出错率低 | 40-60% |
| 进度可控 | 任务列表清晰，随时知道进度 | - |

### 最佳实践

**requirements.md 模板**
```markdown
# [功能名称] 需求文档

## 用户故事
作为 [角色]，我希望 [功能]，以便 [价值]。

## 验收标准
- [ ] 标准 1
- [ ] 标准 2

## 优先级
P0 - 必须完成
```

**design.md 模板**
```markdown
# [功能名称] 设计文档

## 架构决策
[关键决策和理由]

## API 接口
POST /api/[path]
- Request: { field: type }
- Response: { field: type }

## 数据模型
Model { field: type }
```

**tasks.md 模板**
```markdown
# [功能名称] 任务列表

## Task 1: [任务组名称]

### 1.1 [子任务] [2h] [P0] 📋
- 文件：[路径]
- 依赖：无
- 验收：[标准]
```

<summary><b>4. 错误恢复策略</b>（点击展开）</summary>

### 常见错误类型

**类型 1：理解错误（AI 误解需求）**
```
请忽略上一个回答。

让我重新描述需求：
[用更清晰的语言重新描述]

关键点：
1. [关键点 1]
2. [关键点 2]
```

**类型 2：实现错误（代码有 bug）**
```
这段代码有问题：[具体指出问题]

请只修复这个问题，其他部分保持不变。

修复后的代码应该：[明确的验收标准]
```

**类型 3：格式错误（输出格式不对）**
```
输出格式不对。

请重新输出，只包含：
1. 修改的函数（不要输出整个文件）
2. 改动说明（不超过 100 字）
```

**类型 4：上下文丢失（AI 忘记之前的内容）**
```
请回顾我们之前的讨论：

已完成：
1. [已完成的任务 1]
2. [已完成的任务 2]

当前任务：[当前要做的事]
约束：[重要的约束条件]
```

<summary><b>5. 多轮对话管理</b>（点击展开）</summary>

### 对话生命周期

```
开始 → 探索 → 执行 → 验证 → 结束
 ↓      ↓      ↓      ↓      ↓
1-2轮  3-5轮  6-8轮  9-10轮  总结
```

### 各阶段策略

**阶段 1：开始（1-2 轮）**
- 使用结构化 Prompt
- 明确输出格式
- 设定约束条件

**阶段 2：探索（3-5 轮）**
- 渐进式添加上下文
- 针对性提问
- 及时纠错

**阶段 3：执行（6-8 轮）**
- 每轮只做一件事
- 要求只输出改动
- 及时验证结果

**阶段 4：验证（9-10 轮）**
- 检查边界情况
- 补充文档
- 优化代码

**阶段 5：结束（总结）**
```
请用 200 字总结我们完成的工作：
1. 实现了哪些功能
2. 做了哪些关键决策
3. 还有哪些待办事项
```

### 🚨 对话超过 10 轮的处理

1. **生成总结**（推荐）
2. **开新会话**（复制总结）
3. **使用 Spec 工作流**（整理到 Spec 文件）

<summary><b>6. 实战案例</b>（点击展开）</summary>

### 案例 1：修复 Bug

**传统方式（低效）**
```
第 1 轮：粘贴 1000 行代码
第 2 轮：粘贴 200 行日志
第 3 轮：粘贴 300 行 schema
...
总计：10+ 轮，未解决
```

**渐进式（高效）**
```
第 1 轮："登录接口返回 500，预期 200"
第 2 轮："错误：AttributeError: 'NoneType'"
第 3 轮："相关代码：user = db.query(...).first()"
第 4 轮："完美，解决了"
总计：4 轮
```

### 案例 2：实现新功能

**传统方式（低效）**
```
第 1 轮："帮我实现用户认证功能，包括注册、登录、登出……"
[AI 输出 500 行代码]
第 2 轮："这个代码有问题……"
[AI 重新输出 500 行代码]
...
总计：15 轮，代码质量不稳定
```

**Spec 工作流（高效）**
```
第 1 轮："请为用户认证功能创建 Spec 三件套"
第 2 轮："请执行 Task 1.1：实现登录接口"
第 3 轮："Task 1.1 完成，请更新 tasks.md"
第 4 轮："请执行 Task 1.2：添加单元测试"
...
总计：8 轮，节省 70% Token，代码质量稳定
```

### 案例 3：代码重构

**传统方式（低效）**
```
第 1 轮："帮我重构这个文件……（粘贴 500 行）"
[AI 输出 500 行代码]
第 2 轮："改动太大了，我只想优化性能"
[AI 重新输出 500 行代码]
...
总计：多轮，改动不可控
```

**明确约束（高效）**
```
第 1 轮："请优化这个文件的性能，只关注数据库查询部分
约束：
- 只修改数据库查询相关的代码
- 不改变函数签名
- 输出格式：只返回修改的函数"

[AI 只输出 2-3 个函数，约 50 行]

第 2 轮："很好，请添加缓存层"
[AI 只输出缓存代码，约 20 行]

总计：2 轮，改动精准可控
```
