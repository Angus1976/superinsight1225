# 生产上线综合测试计划表（2026）

**目的**：在 `[测试检查清单.md](./测试检查清单.md)`（Docker 全栈 + 种子数据 + 脚本验证）之上，用**多角色、多方法、分层门禁**覆盖上线前风险，避免同一内容被同一手段重复测三遍，同时保证安全稳定可运营。

**真源**：自动化以仓库脚本与 CI（`.github/workflows/commit-tests.yml`）为准；手工项仅在**预发布/生产等价环境**执行并留痕。

**延后与多方（标记约定）**：下列事项依赖**类生产环境、运维 / 测试 / 组织管理员**等协同，**不阻塞**当前仓库侧自动化门禁；正文、章节标题与 §8 留痕中以 **`【延后·多方】`** 标示，条件具备后再执行并补录。

**分支策略（仓库约定）**


| 分支                    | 用途                                    | 合并方式                                             |
| --------------------- | ------------------------------------- | ------------------------------------------------ |
| **main**              | 发布就绪 / 稳定线；与对外版本、生产或预发布标签对齐           | **PR + 审查 + CI 绿**；紧急修复可用 `hotfix/*` → PR → main |
| **develop**           | 日常集成；功能聚合后再合入 main                    | **PR**（默认目标分支）；禁止 force-push（除非团队临时约定）           |
| **feature/**、**fix/** | 从 **develop** 拉出，完成后 PR 回 **develop** | 小步合并，避免长寿命大分支                                    |


**与 CI 的对应关系**：向 **main / develop** 的 push（及指向二者的 PR）跑 `commit-tests` / `pr-tests`（以 workflow `on` 为准）。**merge 到 main** 后跑 `main-branch-tests.yml`（含全量 Chromium E2E）。**不再**配置每日定时全量 E2E；**develop** 的定时回归由 `**e2e-sample-phased.yml`**（每周一 UTC）承担。若需与 main 同规模、含后端的全量 Chromium，可**手动**运行 `**e2e-chromium-scheduled.yml`**（默认 ref **develop**，可改分支或 SHA）。

**默认分支**：建议将 GitHub **default branch** 设为 **main**（文档与模板入口）；定时任务读取的是默认分支上的 workflow 文件，检出代码仍可为 **develop**（本仓库已按上表配置）。

**首次启用 develop**（远程尚无 `develop` 时，依赖检出 **develop** 的 workflow——如 `**e2e-sample-phased.yml` 的定时任务**——会 checkout 失败）：在仓库根执行 `**bash scripts/git-bootstrap-develop.sh`**（从 `origin/main` 创建并 `git push -u origin develop`）。也可手动：`git fetch origin main && git branch develop origin/main && git push -u origin develop`。GitHub 无「PR 默认目标 = develop」的单独全局开关；**协作约定**：日常 PR **base 选 develop**，热修/发版再 **base 选 main**（见 `.github/pull_request_template.md` 提示）。

**发版后同步**：将 `main` 上的发布合并结果并回 `**develop`** 时，维护者可执行 `**bash scripts/git-sync-develop-from-main.sh**`（工作区须干净；有冲突则本地解决后再 `git push`）。

## 【延后·多方】10. 附录：GitHub 组织/仓库规则（分支保护与必填检查）

**当前策略：暂缓实施（【延后·多方】，需具备 GitHub 权限的维护者）。** 不在 GitHub 上强制 Rulesets/分支保护亦可继续开发与合并；需要门禁时再按本节在网页或 CLI 开启。

### 10.1 能否由他人或 AI「代劳」配置？

**不能代劳。** 组织规则、Rulesets、分支保护、必填 Status check 均需在 **GitHub 上具备权限的账号**（组织 **Owner**、具备 **admin** 的仓库维护者，或已安装且授权的 GitHub App）在网页或通过 **已登录本地的 `[gh` CLI](https://cli.github.com/)** / API 完成。AI 无法替你登录 GitHub、也无法在未提供令牌的情况下调用组织级 API。

### 10.2 组织级规则（Organization rulesets）怎么开？

适用于：**同一套规则要套在组织内多个仓库上**、或希望由组织统一强制（需组织权限与对应套餐能力，以 GitHub 当前产品为准）。

1. 使用 **Owner** 账号登录 GitHub。
2. 进入组织：**Settings**（组织设置，非某个仓库）。
3. 在 **Code, planning, and automation** 下找到 **Repository** → **Repository rules**（或 **Rulesets** 相关入口，界面文案可能随版本微调）。
4. **New ruleset** → 选择作用范围（例如 **Selected repositories** 勾选本仓库，或 **All repositories**）。
5. 配置 **Target branches**（例如 `main`、`develop` 或 `refs/heads/main` 等）。
6. 勾选需要的 **Rules**（见下节「建议」）。
7. 保存；必要时在 **Enforcement** 中选 **Active**（可先 **Evaluate** 观察一段时间）。

官方说明：[Managing rulesets for repositories in your organization](https://docs.github.com/en/organizations/managing-organization-settings/managing-rulesets-for-repositories-in-your-organization)

### 10.3 仅单个仓库时（Repository rulesets）

无组织统一要求时，在仓库内即可：

**仓库 → Settings → Rules → Rulesets → New ruleset**（或 **Branch protection rules** 经典界面，与 Rulesets 二选一即可，避免重复冲突）。

官方说明：[About rulesets](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-rulesets/about-rulesets)

### 10.4 与本仓库分支策略对齐的**建议**（按需裁剪）


| 目标分支        | 建议规则                                                                               | 说明                                                      |
| ----------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------- |
| **main**    | 禁止直接 push；**要求 PR**；建议 **Require approvals**（≥1）；**Require status checks to pass** | 合并进 main 会触发 `main-branch-tests.yml`（含 E2E），可与「合并前检查」互补 |
| **develop** | 禁止直接 push；要求 PR；**Require status checks**（至少 `commit-tests` / `pr-tests` 相关 job 绿） | 与日常集成线一致；审批人数可按团队缩小                                     |
| 二者          | **Require linear history**（可选）；**Require conversation resolution**（可选）             | 减少合并噪声                                                  |


**必填 Status checks 名称**：以仓库 **Actions** 里**第一次跑绿**后、PR 页面 **Checks** 里显示的**精确名称为准**（可含 `job / workflow` 前缀）。本仓库常见来源包括：`commit-tests.yml`（如 Backend Unit Tests、Frontend Unit Tests、Quick Validation、Postgres Backup/Restore Drill 等）、`pr-tests.yml` 的 Merge Gate 等。在规则里勾选时务必与列表**一字不差**。

### 10.5 用命令行（可选，仍需你自己登录）

安装并登录：`gh auth login`（需对目标组织/仓库有足够权限）。

查看或管理 rulesets 需用 [GitHub REST API](https://docs.github.com/en/rest/orgs/rules?apiVersion=2022-11-28)（组织级）或 [仓库 rulesets API](https://docs.github.com/en/rest/repos/rules?apiVersion=2022-11-28)。具体 `gh api` 路径随规则类型而异，**首次仍建议在网页创建一条规则**，再在 API 文档中对照 JSON 结构做批量或 IaC。
