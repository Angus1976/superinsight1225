# Kiro Spec 工作流快速指南

## 核心理念

**观全局，做细节**：通过 Spec 文件让 AI 理解完整需求，但每次只执行一个小任务。

---

## Spec 三件套

```
.kiro/specs/{feature-name}/
├── requirements.md  # 需求文档（< 5000 字）
├── design.md        # 设计文档（< 3000 字）
└── tasks.md         # 任务列表（< 2000 字）
```

---

## 快速开始

### 步骤 1: 创建 Spec（观全局）

```
请为 [功能名称] 创建 Spec 三件套：

需求描述：
[用户故事、验收标准、优先级]

约束：
- requirements.md < 5000 字
- design.md < 3000 字
- tasks.md < 2000 字
- 任务可独立验证
```

### 步骤 2: 执行任务（做细节）

```
请执行 tasks.md 中的 Task [编号]：[任务名称]

上下文：
- 需求：#File:.kiro/specs/[feature-name]/requirements.md
- 设计：#File:.kiro/specs/[feature-name]/design.md
- 当前任务：tasks.md 第 [行号] 行

约束：
- 只实现这一个任务
- 遵循 design.md 中的设计
- 不要修改其他代码
- 输出格式：只返回新增/修改的代码
```

### 步骤 3: 更新进度

```
Task [编号] 已完成，请更新 tasks.md：
- 标记为完成 ✅
- 添加实际耗时
- 如果有新问题，添加到待办

只输出更新后的 tasks.md 相关部分。
```

## 模板

### requirements.md 模板

```markdown
# [功能名称] 需求文档

## 用户故事
作为 [角色]，我希望 [功能]，以便 [价值]。

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 优先级
P0 - 必须完成

## 约束条件
- 约束 1
- 约束 2

## 非功能需求
- 性能：响应时间 < 200ms
- 安全：使用 JWT 认证
```

### design.md 模板

```markdown
# [功能名称] 设计文档

## 架构决策
[关键的架构决策和理由]

## API 接口
### [接口名称]
- Method: POST
- Path: /api/[path]
- Request: { field: type }
- Response: { field: type }

## 数据模型
### [模型名称]
```
Model {
  field1: type
  field2: type
}
```

## 关键流程
1. 步骤 1
2. 步骤 2
3. 步骤 3
```

### tasks.md 模板

```markdown
# [功能名称] 任务列表

## Task 1: [任务组名称]

### 1.1 [子任务名称] [预估时间] [状态]
- 文件：[文件路径]
- 依赖：[依赖的任务编号]
- 验收：[验收标准]

### 1.2 [子任务名称] [预估时间] [状态]
- 文件：[文件路径]
- 依赖：[依赖的任务编号]
- 验收：[验收标准]

## Task 2: [任务组名称]

### 2.1 [子任务名称] [预估时间] [状态]
- 文件：[文件路径]
- 依赖：[依赖的任务编号]
- 验收：[验收标准]

---

## 状态说明
- 📋 待办
- ⏳ 进行中
- ✅ 已完成
- ❌ 已取消
```

## 优势对比

| 方式 | 上下文大小 | Token 使用 | AI 准确率 | 可控性 |
|------|-----------|-----------|----------|--------|
| **传统聊天** | 5000+ tokens | 高 | 中 | 低 |
| **Spec 工作流** | 500-1000 tokens | 低（节省 70-80%） | 高 | 高 |

## 检查清单

使用 Spec 工作流时，确保：

- [ ] 创建了 Spec 三件套
- [ ] requirements.md < 5000 字
- [ ] design.md < 3000 字
- [ ] tasks.md < 2000 字
- [ ] 每个任务可独立验证
- [ ] 任务之间的依赖关系清晰
- [ ] 执行任务时只引用必要的 Spec 文件
- [ ] 每次只执行一个任务
- [ ] 任务完成后更新 tasks.md

## 常见问题

### Q: 什么时候使用 Spec 工作流？

**A**: 当任务满足以下条件时：
- 需要多个步骤完成
- 涉及多个文件
- 需要保持全局视野
- 希望控制开发节奏

### Q: Spec 文件太大怎么办？

**A**: 拆分功能：
- 大功能拆成多个小功能
- 每个小功能一个 Spec 目录
- 在 requirements.md 中引用其他 Spec

### Q: 任务执行顺序如何确定？

**A**: 按依赖关系：
- 先执行无依赖的任务
- 再执行依赖已完成任务的任务
- tasks.md 中明确标注依赖关系
