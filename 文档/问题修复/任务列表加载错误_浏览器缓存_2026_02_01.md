# 任务列表加载错误 - 浏览器缓存问题

**日期**: 2026-02-01  
**状态**: ✅ 已识别  
**优先级**: HIGH  
**问题类型**: 浏览器缓存

## 问题描述

用户访问任务列表页面时，出现 400 Bad Request 错误，无法加载任务列表。

### 错误信息

```
GET /api/tasks/list HTTP/1.1 400 Bad Request
```

### 浏览器控制台错误

```
Failed to load resource: the server responded with a status of 400 (Bad Request)
```

## 根本原因

**浏览器缓存了旧版本的 JavaScript 代码**

### 详细分析

1. **历史背景**:
   - 之前的代码版本使用 `/api/tasks/list` 端点
   - 最新代码已更新为 `/api/tasks` 端点
   - 前端容器已重建，代码已更新

2. **当前状态**:
   - 后端只定义了 `/api/tasks` 端点（正确）
   - 前端代码使用 `/api/tasks` 端点（正确）
   - 但浏览器缓存了旧的 JavaScript 文件，仍在请求 `/api/tasks/list`

3. **验证方法**:
   ```bash
   # 检查后端日志
   docker compose logs app | grep "tasks/list"
   # 输出: GET /api/tasks/list HTTP/1.1 400 Bad Request
   
   # 检查前端代码
   grep -r "tasks/list" frontend/src/
   # 输出: (无匹配) - 代码中已不存在此端点
   ```

## 解决方案

### 方案 1: 硬刷新浏览器（推荐）

**适用场景**: 单个用户遇到问题

**操作步骤**:

1. **macOS / Linux**:
   - Chrome/Edge: `Cmd + Shift + R`
   - Firefox: `Cmd + Shift + R`
   - Safari: `Cmd + Option + R`

2. **Windows**:
   - Chrome/Edge: `Ctrl + Shift + R` 或 `Ctrl + F5`
   - Firefox: `Ctrl + Shift + R` 或 `Ctrl + F5`

3. **验证**:
   - 打开浏览器开发者工具 (F12)
   - 切换到 Network 标签
   - 刷新页面
   - 检查请求是否变为 `/api/tasks` (不带 `/list`)

### 方案 2: 清除浏览器缓存

**适用场景**: 硬刷新无效时

**Chrome/Edge 操作步骤**:

1. 打开开发者工具 (F12)
2. 右键点击刷新按钮
3. 选择"清空缓存并硬性重新加载"

**或者**:

1. 打开设置 (Chrome: `chrome://settings/`)
2. 搜索"清除浏览数据"
3. 选择"缓存的图片和文件"
4. 点击"清除数据"

**Firefox 操作步骤**:

1. 打开设置 (`about:preferences`)
2. 隐私与安全 → Cookie 和网站数据
3. 点击"清除数据"
4. 勾选"缓存的 Web 内容"
5. 点击"清除"

### 方案 3: 使用无痕/隐私模式

**适用场景**: 快速验证是否为缓存问题

**操作步骤**:

1. 打开无痕窗口:
   - Chrome/Edge: `Cmd + Shift + N` (macOS) 或 `Ctrl + Shift + N` (Windows)
   - Firefox: `Cmd + Shift + P` (macOS) 或 `Ctrl + Shift + P` (Windows)
   - Safari: `Cmd + Shift + N`

2. 访问应用: `http://localhost:5173`

3. 如果无痕模式下正常，确认是缓存问题

### 方案 4: 禁用缓存（开发模式）

**适用场景**: 开发和测试期间

**Chrome/Edge 操作步骤**:

1. 打开开发者工具 (F12)
2. 切换到 Network 标签
3. 勾选"Disable cache"
4. 保持开发者工具打开状态

**Firefox 操作步骤**:

1. 打开开发者工具 (F12)
2. 点击设置图标（齿轮）
3. 勾选"禁用 HTTP 缓存（当工具箱打开时）"

### 方案 5: 添加版本号到静态资源（长期方案）

**适用场景**: 防止未来出现类似问题

**实现方法**:

在 `frontend/vite.config.ts` 中配置:

```typescript
export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        // 添加哈希值到文件名
        entryFileNames: 'assets/[name].[hash].js',
        chunkFileNames: 'assets/[name].[hash].js',
        assetFileNames: 'assets/[name].[hash].[ext]'
      }
    }
  }
});
```

这样每次构建都会生成新的文件名，浏览器会自动加载新版本。

## 预防措施

### 1. 开发阶段

- 在开发者工具中启用"Disable cache"
- 使用无痕模式测试
- 定期清除浏览器缓存

### 2. 部署阶段

- 使用版本号或哈希值命名静态资源
- 配置适当的 Cache-Control 头
- 在 nginx/CDN 层面配置缓存策略

### 3. 用户通知

- 在重大更新后通知用户清除缓存
- 提供清除缓存的操作指南
- 在应用中添加版本检测和提示

## 相关配置

### Vite 缓存配置

```typescript
// frontend/vite.config.ts
export default defineConfig({
  server: {
    headers: {
      'Cache-Control': 'no-store'  // 开发环境禁用缓存
    }
  },
  build: {
    rollupOptions: {
      output: {
        // 生产环境使用哈希
        entryFileNames: 'assets/[name].[hash].js',
        chunkFileNames: 'assets/[name].[hash].js',
        assetFileNames: 'assets/[name].[hash].[ext]'
      }
    }
  }
});
```

### Nginx 缓存配置

```nginx
# 对于 HTML 文件，不缓存
location / {
    add_header Cache-Control "no-cache, no-store, must-revalidate";
    add_header Pragma "no-cache";
    add_header Expires "0";
}

# 对于带哈希的静态资源，长期缓存
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
    add_header Cache-Control "public, max-age=31536000, immutable";
}
```

## 常见问题

### Q1: 为什么硬刷新后还是有问题？

**A**: 可能的原因:
1. Service Worker 缓存了旧版本
2. CDN 或代理服务器缓存
3. 浏览器扩展干扰

**解决方法**:
1. 清除 Service Worker: 开发者工具 → Application → Service Workers → Unregister
2. 禁用浏览器扩展
3. 使用无痕模式测试

### Q2: 如何确认是否为缓存问题？

**A**: 检查方法:
1. 在 Network 标签中查看请求的 URL
2. 检查响应头中的 `Cache-Control`
3. 对比无痕模式和正常模式的行为

### Q3: 为什么其他用户没有这个问题？

**A**: 可能的原因:
1. 其他用户首次访问，没有旧缓存
2. 其他用户使用了不同的浏览器
3. 其他用户的浏览器缓存策略不同

## 技术细节

### 浏览器缓存机制

1. **内存缓存 (Memory Cache)**:
   - 最快，但关闭标签页后清除
   - 硬刷新会绕过

2. **磁盘缓存 (Disk Cache)**:
   - 持久化，关闭浏览器后仍存在
   - 需要清除缓存才能删除

3. **Service Worker 缓存**:
   - 独立于浏览器缓存
   - 需要单独清除

### HTTP 缓存头

```http
# 不缓存
Cache-Control: no-cache, no-store, must-revalidate
Pragma: no-cache
Expires: 0

# 长期缓存
Cache-Control: public, max-age=31536000, immutable
```
