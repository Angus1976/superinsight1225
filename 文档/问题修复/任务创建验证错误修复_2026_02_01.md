# 任务创建验证错误修复

**日期**: 2026-02-01  
**状态**: ✅ 已修复  
**优先级**: HIGH

## 问题描述

用户在创建新任务时遇到验证错误，无法保存任务。后端日志显示以下验证错误：

```json
{
  "type": "missing",
  "loc": ["body", "name"],
  "msg": "Field required"
}
{
  "type": "missing",
  "loc": ["body", "data_source", "type"],
  "msg": "Field required"
}
```

## 根本原因

### 1. 前端问题

**TaskCreateModal.tsx** 中的 `handleSubmit` 函数存在以下问题：

1. **缺少 `name` 字段**: 虽然表单中有 `name` 输入框，但在构建 payload 时正确包含了该字段
2. **`data_source` 对象不完整**: 前端总是发送 `data_source` 对象，即使用户没有选择数据源，导致发送了不完整的对象：
   ```typescript
   data_source: {
     type: values.data_source_type,  // 可能为 undefined
     config: {
       source_id: values.data_source_id,  // 可能为 undefined
       // ...
     }
   }
   ```

### 2. 后端验证

**src/api/tasks.py** 中的 `TaskCreateRequest` 模型定义：

```python
class DataSourceConfig(BaseModel):
    type: str = Field(..., description="Data source type")  # 必填
    config: Dict[str, Any] = Field(default_factory=dict)

class TaskCreateRequest(BaseModel):
    name: str = Field(..., description="Task name")  # 必填
    data_source: Optional[DataSourceConfig] = Field(None)  # 可选
```

问题：
- `name` 字段是必填的 ✅
- `data_source` 是可选的 ✅
- 但如果提供了 `data_source`，则 `type` 字段是必填的 ❌

## 修复方案

### 1. 前端修复 (frontend/src/pages/Tasks/TaskCreateModal.tsx)

修改 `handleSubmit` 函数，只在用户实际选择了数据源时才发送 `data_source` 对象：

```typescript
const handleSubmit = async () => {
  try {
    const values = await form.validateFields();
    
    // 只在用户选择了数据源时才构建 data_source 对象
    let dataSource = undefined;
    if (values.data_source_type && values.data_source_id) {
      dataSource = {
        type: values.data_source_type,
        config: {
          source_id: values.data_source_id,
          file_path: values.file_path,
          api_endpoint: values.api_endpoint,
          database_query: values.database_query,
          ...annotationConfig,
        },
      };
    }
    
    const payload: CreateTaskPayload = {
      name: values.name,
      description: values.description,
      priority: values.priority,
      annotation_type: values.annotation_type,
      assignee_id: values.assignee_id,
      due_date: values.due_date?.toISOString(),
      tags: values.tags,
      data_source: dataSource,  // 可能为 undefined
    };

    await createTask.mutateAsync(payload);
    // ...
  }
};
```

**关键改进**:
- 检查 `data_source_type` 和 `data_source_id` 是否存在
- 只有在两者都存在时才构建 `data_source` 对象
- 否则 `data_source` 为 `undefined`，不会发送到后端

### 2. 后端改进 (src/api/tasks.py)

为 `DataSourceConfig` 添加配置，允许额外字段：

```python
class DataSourceConfig(BaseModel):
    """Data source configuration for task"""
    type: str = Field(..., description="Data source type: file, api, or database")
    config: Dict[str, Any] = Field(default_factory=dict, description="Data source configuration")
    
    class Config:
        extra = "allow"  # 允许额外字段
```

**改进点**:
- 添加 `Config` 类，设置 `extra = "allow"`
- 允许客户端发送额外的配置字段，提高灵活性

## 经验教训

### 1. 前端表单验证

**问题**: 前端总是发送可选字段，即使用户没有填写

**最佳实践**:
- 在构建 payload 前检查字段是否有值
- 只发送用户实际填写的字段
- 使用 `undefined` 而不是空对象表示未填写

### 2. 后端验证

**问题**: 可选字段的嵌套必填字段导致验证混乱

**最佳实践**:
- 如果整个对象是可选的，不要在嵌套字段上使用 `Field(...)`
- 使用 `Field(None)` 或 `Field(default=...)`
- 在业务逻辑中验证必填字段，而不是在模型层

### 3. 错误调试

**有效方法**:
- 查看后端日志中的详细验证错误
- 使用浏览器开发者工具查看实际发送的请求
- 对比前端发送的数据和后端期望的数据结构
