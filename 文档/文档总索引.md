## 🤖 AI 与工程可溯性

**目的**：让 AI 与开发者在任意时刻都能从**固定入口**恢复对仓库结构、规则与进度的可靠理解；用**可追溯**的规则与变更索引减少「对话失忆」；用明确**真源**降低文档与代码长期不一致。


| 需求                                   | 建议入口                                                         |
| ------------------------------------ | ------------------------------------------------------------ |
| 默认原则与何时读哪条规则                         | `.kiro/steering/CORE.md`                                     |
| 项目结构 / 技术栈 / 产品（全局鸟瞰）                | `.kiro/rules/structure.md`、`tech.md`、`product.md` + 下方「全局分析」 |
| 进度与里程碑（可能与分支不一致，需交叉验证）               | `.kiro/project-progress.json`                                |
| `文档/` 树导航与本页                         | **本文件**（`文档/文档总索引.md`）                                       |
| 长记忆 vs 短记忆、代码与文档对齐约定、工程变更简表          | [工程可溯性与AI上下文.md](./索引和说明/工程可溯性与AI上下文.md)                     |
| Cursor 侧 KIRO 摘要（与 CORE 冲突以 CORE 为准） | `.cursor/skills/superdata-kiro-dev/SKILL.md`                 |


**一致性（必读摘要）**：

- **运行时行为以代码为准**；`.kiro/specs/` 三件套与本文档类材料可能滞后，重大结论需与实现核对。
- **避免出现**「文档已设计功能但未标注未实现」或「代码已改文档未改」：改代码应同步更新对应 Spec 三件套或专题文档状态；改设计应先评估实现再落文档。
- **禁止**为可溯性单独堆砌报告类长文；以索引、三件套、Git 记录与 [工程可溯性与AI上下文.md](./索引和说明/工程可溯性与AI上下文.md) 中的简表为主。

**全局分析（与用户提到「项目全貌 / 全局分析」等时）**：按 CORE 建议一次性结合 `product`、`structure`、`tech`、`project-progress.json` 与本索引；细节见工程可溯性文档与 `.kiro/steering/CORE.md`。
